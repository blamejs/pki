# Roadmap

`@blamejs/pki` is a pure-JavaScript (CommonJS) PKI toolkit: X.509, CMS, OCSP, timestamping, PKCS enrollment and certificate formats, built post-quantum-first with no npm runtime dependencies and security defaults that are wired in rather than opt-in. It reimplements the standards surface of the classical PKI.js toolkit while owning its own ASN.1 codec and crypto layer, so correctness, denial-of-service resistance, and algorithm agility are in-tree rather than inherited from external parsers or the Web Crypto ceiling.

This document tracks capability areas and the standards they target, in rough delivery order. Every item is marked **Shipped** (available in the current release), **Targeted** (committed for a near-term release), **Planned** (accepted, scheduled later), or **Under design** (shape still being settled). Status changes as capabilities ship; consult the changelog for what is released today.

## Foundations

The parser and algorithm layers are the load-bearing decisions the rest of the library rides on.

- **Strict, validating DER/BER codec** — *Shipped.* An in-house, purpose-built ASN.1 encoder/decoder with a hard depth and size bound so malformed or deeply nested input is rejected in bounded time instead of exhausting the stack. DER is canonical on encode; the decoder rejects non-canonical structures by default and accepts BER only where a specific standard permits it. A single structure definition drives both encode and decode so context-tag (EXPLICIT/IMPLICIT) handling cannot diverge between the two directions.
- **Data-driven algorithm registry** — *Shipped.* The OID registry (`pki.oid`) names every algorithm, attribute, and extension in both directions, and the resolution table maps an OID to its parameters, key importer, signer, and verifier — so a signature or KEM algorithm, including the post-quantum sets, is a registry entry rather than a special case across signing, verification, and path validation.
- **Node-native crypto engine** — *Shipped.* Cryptographic operations run over Node's native `node:crypto` through the W3C WebCrypto engine (`pki.webcrypto`), which removes the Web Crypto limits on streaming, opaque (non-extractable) key handles, and algorithm coverage. It ships the full classical set plus post-quantum ML-DSA and SLH-DSA signatures and ML-KEM key generation, encoding, and import; KEM encapsulation lands with the CMS KEM-decrypt surface.
- **Fail-closed operation** — *Shipped.* Every decode and (as they land) encrypt, sign, and verify path throws on failure through the typed `PkiError` taxonomy. No path emits zero, default, or partial output in place of a real result.

## X.509 certificates and CRLs — RFC 5280

- **Certificate parse / build / sign / verify** — *Parsing and signing shipped.* `pki.schema.x509.parse` decodes a v3 certificate, and `pki.x509.sign(spec, issuer, opts)` builds and signs one — self-signed (a key alone) or CA-signed (an explicit issuer name and public key, or an issuing certificate) — returning DER or a PEM `CERTIFICATE`. The signature algorithm is resolved from the signing key through the shared registry, so RSA (PKCS#1 v1.5 / PSS), ECDSA on P-256/P-384/P-521, Ed25519, Ed448, ML-DSA-44/65/87, the twelve SLH-DSA sets, and the composite (hybrid) arms are all available. It encodes the extension set from the spec — basic constraints, key usage and extended key usage, subject and authority key identifiers, subject alternative names, and certificate policies — with any other extension supplied as pre-encoded DER; derives the version from the field set (v3 with extensions, else v1); and enforces the RFC 5280 serial-number bounds, the UTCTime/GeneralizedTime cutover, the DER default omissions (v1 tag, false criticality, false cA), and the CA cross-field rules (keyCertSign and path length require cA). Every emitted certificate round-trips through the parser and is independently verified by OpenSSL across every algorithm arm. The remaining standard extension encoders (name constraints, policy constraints/mappings, CRL distribution points, authority/subject information access) land natively as the demand appears; the pre-encoded-DER form covers them now. RFC 5280 / RFC 9909 (SLH-DSA) / RFC 9882 (ML-DSA).
- **Distinguished names as ordered RDN collections** — *Shipped.* Names are modeled as an ordered sequence of relative distinguished names, each an unordered set of type/value pairs, so multi-valued RDNs parse, encode, and compare under the RFC 5280 §7.1 canonical DN equality across every format.
- **CRLs** — *Shipped.* `pki.schema.crl.parse` decodes a certificate revocation list and `pki.path.crlChecker` performs revoked-certificate lookup with signature, issuer-authorization, distribution-point-scope, and currency checks; `pki.crl.sign` builds and signs a `CertificateList` (RFC 5280 §5) over any registry algorithm — the version derived from the extension set, an empty revocation list omitting the field, `reasonCode` an ENUMERATED and `invalidityDate` always GeneralizedTime, the CRL/entry extensions (authority key identifier, cRLNumber, issuing distribution point, delta-CRL indicator, freshest CRL, authority information access, reason code, invalidity date, certificate issuer) with their RFC-fixed criticality — while `pki.crl.verify` checks a CRL signature through the one path-validation signature engine and `pki.crl.isRevoked` looks a serial up. Cross-checked against OpenSSL.
- **Human-readable inspection (`openssl x509/crl/req/cms -text` in pure JS)** — *Shipped.* `pki.inspect.certificate` / `.crl` / `.csr` / `.cms` turn a certificate, certificate revocation list, PKCS#10 request, or CMS message (PEM, DER, or a parsed result) into the familiar OpenSSL-style report, and `pki.inspect.any` detects the format and routes to the right one — so an operator reads a PEM or DER blob without shelling out to OpenSSL. A certificate report shows version, serial, signature algorithm, the issuer and subject distinguished names, the validity window, the public-key details, and every decoded extension; a CRL shows the issuer, Last/Next Update, the CRL extensions, and each revoked entry with its serial, revocation date, and named reason code; a CSR shows the subject, the public key, and the requested extensions and attributes; a CMS SignedData shows the content type, digest algorithms, embedded certificates, and each SignerInfo with its signer identifier, algorithms, signed/unsigned attributes, and signature (a non-SignedData ContentInfo gets a stable top-field summary). All four compose one set of field renderers over the strict parsers, the shared extension decoders, and the two-way OID registry — so they name the same vocabulary the verifier resolves, name OIDs an OpenSSL build shows only as raw octets, and never drift; an unregistered OID renders as its dotted string and an undecodable value as raw octets. Best-effort: a malformed part hex-dumps rather than sinking the report, and only entry-point coercion throws. The format is stable and OpenSSL-familiar rather than pinned to one OpenSSL version; verified field-for-field against `openssl x509/req/cms -text`. `pki.schema.detectFormat` is the detection companion of `pki.schema.parse`.
- **Qualified certificate statements** — *Shipped.* `pki.schema.x509.parse` decodes the RFC 3739 §3.2.6 / ETSI EN 319 412-5 `qcStatements` extension — the EU qualified-certificate declaration (QcCompliance), reliance limit (QcLimitValue), QSCD-key flag (QcSSCD), certificate-purpose type (QcType), retention period, PDS URLs, country-of-qualification, and the PKIX SemanticsInformation statements — decoded and validated by the toolkit's certificate-extension decoders and rendered by `pki.inspect`, with an unknown statement preserved opaque and every malformed shape failing closed as a typed error. RFC 3739 / ETSI EN 319 412-5.
- **Microsoft enterprise CA extensions** — *Shipped.* `pki.schema.x509.parse` decodes the Active Directory Certificate Services enrollment extensions and `pki.inspect` renders them: the v2 certificate template marker (`szOID-CERTIFICATE_TEMPLATE` — the template OID plus its major/minor version, each a full DWORD), the legacy v1 template name (`szOID-ENROLL_CERTTYPE_EXTENSION`, a BMPString), the CA version (`szOID-CERTSRV_CA_VERSION`, surfaced both as the raw DWORD and split into the CA key index and certificate index), the previous-CA-certificate hash (`szOID-CERTSRV_PREVIOUS_CERT_HASH`), and the application policies (`szOID-APPLICATION_CERT_POLICIES`, decoded with the same rules as RFC 5280 certificate policies). Each fails closed as a typed error on a malformed shape; a critical instance is rejected in certification-path validation and flagged by `pki.lint`. [MS-WCCE] / [MS-CRTD].

## Certification requests — RFC 2986 (PKCS#10)

- **CSR parse / build / sign / verify** — *Parsing and signing shipped.* `pki.schema.csr.parse` decodes a PKCS#10 request, and `pki.csr.sign(spec, key, opts)` builds and signs one — self-signed by the subject's own key for proof of possession (no issuer), over any signature algorithm the registry resolves (RSA PKCS#1 v1.5 / PSS, ECDSA, EdDSA, ML-DSA, SLH-DSA, composite). First-class helpers accept strings and plain objects: a subject (which may be empty), a `challengePassword`, and an `extensionRequest` carrying the requested v3 extensions (subject alternative names, key usage, extended key usage, basic constraints, certificate policies, subject key identifier — or pre-encoded Extension DER) that a CA copies into the issued certificate. The produced request's proof-of-possession signature is verified against the subject key before it is returned (what `openssl req -verify` checks), and every emitted request round-trips through the parser and OpenSSL. RFC 2986 / RFC 2985.

## Certification path validation — RFC 5280

- **Path validation engine** — *Shipped.* `pki.path.validate` runs the RFC 5280 §6.1 validation algorithm over an ordered path and a trust anchor: signature chaining, validity windows, name chaining, key usage, basic constraints and path length, name constraints, and the certificate-policy tree. Trust anchors may be roots or intermediates.
- **Path building** — *Shipped.* `pki.path.build(leaf, opts)` discovers the ordered leaf→anchor path `pki.path.validate` consumes, from a leaf certificate, an untrusted pool of candidate CA certificates, and a trust store. Candidate issuers are matched by RFC 5280 §7.1 name chaining and prioritized by the RFC 4158 §3.5 heuristics (a subjectKeyIdentifier/authorityKeyIdentifier match, an anchor-adjacent issuer, CA + keyCertSign, validity at the check time — all ordering hints, never filters), then searched depth-first with backtracking so the first path `validate` accepts wins; a name or key-identifier match is only a hint, and every accept flows through `validate`, so building never weakens or duplicates a §6.1 check. The search over the untrusted pool is bounded — a chain-length cap, a total-work cap on candidate expansions, and a visited-set keyed on the certificate identity tuple — so a cross-certificate cycle or Bridge-CA fan-out terminates deterministically; the trust store accepts anchor tuples or self-signed root certificates, `opts.validate:false` returns the ordered path unvalidated, and the accept/reject verdict is cross-checked against `openssl verify`. AIA `caIssuers` network fetching (an opt-in async fetcher), reverse/bridge-mesh building, and multiple-anchor policy selection remain *Planned*. RFC 4158 / RFC 5280 §6.
- **Structured results** — *Shipped.* Validation returns the complete path plus a per-check reason code for every step. Validity-window enforcement is always on, with the check date an explicit input. Validation is pure and re-entrant — input certificate state is never mutated, so a chain can be validated repeatedly.
- **Revocation checking** — *Shipped (CRL + OCSP).* A pluggable revocation hook is integrated into the path result. `pki.path.crlChecker` consults CRLs (signature, issuer authorization, distribution-point scope, currency); `pki.path.ocspChecker` consults pre-fetched OCSP responses (RFC 6960) over the same interface — matching the full CertID triple against the certificate, recomputing `issuerNameHash`/`issuerKeyHash` under the CertID's own hash algorithm, confirming the responder is authorized (the issuing CA directly or a CA-issued delegate bearing id-kp-OCSPSigning), verifying the response signature, and enforcing `thisUpdate`/`nextUpdate` currency. Both fail closed to an undetermined status unless the caller opts into soft-fail. RFC 6960.
- **Adversarial conformance corpus** — *Targeted.* Name-constraint and path-validation behavior is gated against a public adversarial corpus.

## Cryptographic Message Syntax — RFC 5652

- **SignedData** — *Signing + verification shipped.* `pki.cms.sign(content, signers, opts)` produces a CMS SignedData (RFC 5652 §5) and `pki.cms.verify(input, opts)` verifies one — both over attached and detached content, single- and multiple-signer, RSA / RSASSA-PSS / ECDSA / EdDSA / ML-DSA (the post-quantum ML-DSA-44/65/87, RFC 9882). Signing builds the signed attributes (content-type, message-digest, signing-time, plus any caller-supplied attribute) as canonical DER, signs the exact §5.4 preimage, and emits a DER `Buffer` or PEM; the signer identifier is issuerAndSerialNumber or subjectKeyIdentifier. Verification locates each signer certificate by its issuerAndSerialNumber or subjectKeyIdentifier (embedded or caller-supplied) and checks the signature over the exact §5.4 preimage: with signed attributes present, the message-digest attribute must equal the content digest and the signature covers the DER re-encoding of the SignedAttributes (the on-wire `[0]` implicit tag replaced by a universal SET OF); otherwise the signature covers the content directly. Each parameter comes from the correct structure — the content digest, the signer's digest algorithm, and the key's own algorithm identifier are kept distinct — and verification reports a per-signer verdict, deferring the trust chain to the caller's `pki.path.validate`. **Countersignatures and unsigned attributes are shipped** — `pki.cms.countersign` adds an RFC 5652 §11.4 countersignature (a `SignerInfo` over the countersigned signature value, any signer algorithm, nestable, the primary bytes preserved) and `pki.cms.verify` returns each countersignature's verdict under `signers[i].countersignatures` and every unsigned attribute (an RFC 3161 timestamp token, attachable via `pki.cms.sign`'s `unsignedAttributes`) under `signers[i].unsignedAttrs`, surfaced unauthenticated. Composite-signature `SignerInfo` signing and verification are shipped (see **Composite (hybrid) signatures**).
- **EnvelopedData with a first-class recipient model** — *Shipped.* `pki.cms.encrypt` / `pki.cms.decrypt` encrypt and decrypt with all five RFC 5652 recipient types: key-transport (RSAES-OAEP-SHA256/384/512, with PKCS#1 v1.5 decrypt-only under the RFC 3218 implicit-rejection countermeasure — never emitted), key-agreement (ephemeral-static ECDH over P-256/384/521 with the ANSI-X9.63 KDF, and X25519/X448 with HKDF per RFC 8418), symmetric key-wrap (AES-128/192/256 key wrap), password (PBKDF2 + the RFC 3211 PWRI-KEK), and the post-quantum ML-KEM KEMRecipientInfo. Recipient identification is pluggable: issuer-and-serial and subject-key-identifier for both directions. One fresh content-encryption key is wrapped for every recipient; content is AES-128/192/256-GCM (an authenticated AuthEnvelopedData) or -CBC. Key agreement follows RFC 5753 / RFC 8418. Both directions cross-check against `openssl cms`.
- **EncryptedData** — *Shipped.* `pki.cms.encrypt` / `pki.cms.decrypt` protect content with a raw predefined key (`{ cek }`) or a PBES2 password (`{ password }`, RFC 8018). RFC 5652 §8.
- **AuthenticatedData** — *Shipped (parse); MAC generation/verification Targeted.* `pki.schema.cms.parse` decodes `id-ct-authData` with the §9.1 version rule, the digestAlgorithm⇔authAttrs biconditional, the content-binding attribute rules, and the raw `authAttrsBytes`/`mac` a verifier consumes.
- **AuthEnvelopedData** — *Shipped.* `pki.cms.encrypt` produces and `pki.cms.decrypt` verifies an `id-ct-authEnvelopedData` with AES-128/192/256-GCM content — the authenticated default. The authenticated attributes are MACed under their EXPLICIT SET OF re-encoding (RFC 5083 §2.2); GCMParameters carry the nonce and a 16-octet ICV matching the mac length (RFC 5084). Integrity is verified before any plaintext is released. RFC 5083 / RFC 5084.
- **KEMRecipientInfo** — *Shipped.* `pki.cms.encrypt` / `pki.cms.decrypt` produce and consume the `id-ori-kem` OtherRecipientInfo arm end to end: ML-KEM-512/768/1024 encapsulation, the `CMSORIforKEMOtherInfo` KDF-info rebuilt identically on both sides, HKDF-SHA256 to the wrap key (512→aes128-wrap, 768/1024→aes256-wrap), with version 0, kekLength⇔wrap consistency, and FIPS 203 ciphertext sizes enforced before decapsulation. RFC 9629 / RFC 9936.
- **CompressedData** — *Shipped.* `pki.cms.compress` / `pki.cms.decompress` produce and consume an `id-ct-compressedData` (RFC 3274): ZLIB (RFC 1950 / RFC 1951) compression, version 0, `id-alg-zlibCompress` with omitted parameters, the RFC 1950 stream as the encapsulated content. Decompress bounds the uncompressed output at `C.LIMITS.COMPRESS_MAX_BYTES` (16 MiB, tightened downward via `opts.maxOutputBytes`) and stops before the output is materialized, so a decompression bomb fails closed as `cms/decompress-too-large` rather than exhausting memory. CompressedData is a size transform only — no integrity, confidentiality, or authentication (RFC 8551 §2.4.5). RFC 3274.
- **Streaming CMS** — *Planned.* Streaming sign/verify and encrypt/decrypt for large payloads, enabled by the in-house crypto engine.

## Keys and credential stores

- **PKCS#8 private keys and SPKI public keys** — *Shipped.* `pki.key.export` / `pki.key.import` move a private key as PKCS#8 or a public key as SubjectPublicKeyInfo, with the algorithm parameters delegated to the encoder so RSA carries an explicit NULL, EC a named curve, and Ed25519/Ed448/X25519/X448 omit parameters; `pki.key.generate` and `pki.key.publicFromPrivate` produce and derive keys over RSA, EC, the Edwards and Montgomery curves, and the post-quantum ML-DSA / ML-KEM. RFC 5958 / RFC 5280.
- **Password-based encryption** — *Shipped.* `pki.key.encrypt` / `pki.key.decrypt` protect a private key as an RFC 5958 EncryptedPrivateKeyInfo under RFC 8018 PBES2 (PBKDF2 + AES-CBC), with UTF-8 password encoding (correct for non-ASCII passwords), byte-exact-with-OpenSSL parameters (a default pseudorandom function and keyLength omitted), salt and iteration counts bounded against a denial-of-service cap before any derivation, and a fail-closed decrypt that is not a padding oracle. PBES1, PBMAC1, and scrypt are refused. RFC 8018.
- **PKCS#12 (PFX)** — *Shipped (parse + build/MAC; bag decryption Targeted).* Parse key-and-certificate stores: authenticated safe, cert/CRL/key/secret bags, shrouded key bags (algorithm surfaced, ciphertext opaque), nested safe contents, `friendlyName` / `localKeyId` attributes, and MAC integrity surfaced with the exact MACed byte range — RFC 9579 PBMAC1 recognized. BER content regions (indefinite lengths, constructed octet strings) accepted exactly where §4.1 requires them; interoperability with OpenSSL is a release acceptance gate. `pki.pkcs12.build` / `pki.pkcs12.verifyMac` / `pki.pkcs12.open` assemble, MAC-verify, and read back a password-integrity store — every bag type, shrouded keys and cert safes encrypted under PBES2, a classic Appendix B HMAC or an RFC 9579 PBMAC1 MAC, with the PKCS#12 BMPString/UTF-8 password split that makes a file open in OpenSSL and NSS (cross-checked bidirectionally). `open` verifies the MAC before decrypting (a wrong password is the MAC verdict), returns the keys as re-validated PKCS#8 DER and the certs/CRLs/secrets as raw DER, and refuses a MAC-less store (unless opted in), a public-key-integrity store, or a legacy non-PBES2 bag. Public-key (signedData) integrity and legacy-PBE (App. C 3DES/RC2) bag decryption remain Targeted. RFC 7292, RFC 9579, RFC 8018.

## Post-quantum and hybrid cryptography

The clearest differentiator: the classical toolkit this library replaces has no post-quantum surface at all. PQC algorithms are registry entries alongside classical ones, not bolt-ons.

- **ML-KEM in X.509** — *Shipped.* ML-KEM public keys in certificates and PKCS#8 private keys: parse and import all three parameter sets, enforce the keyEncipherment-only key usage in path validation, and lint the certificate and encapsulation-key size against the algorithm OID. RFC 9935 / FIPS 203.
- **ML-DSA in X.509** — *Signing shipped.* `pki.x509.sign` issues an ML-DSA-44/65/87 certificate (`parameters` absent, RFC 9882/RFC 9909 algorithm identifiers) — self-signed or CA-signed — and certification-path validation verifies an ML-DSA certificate signature; both tracked against the LAMPS certificate specification through publication. draft-ietf-lamps-dilithium-certificates.
- **SLH-DSA** — *Verification shipped; CMS signing + verification shipped.* All twelve FIPS 205 parameter sets verify in certification-path validation (`pki.path.validate`), and `pki.cms.sign` / `pki.cms.verify` now sign and verify a CMS SignedData with any of the twelve pure sets — pure mode, empty context, the parameter set's message digest pinned per RFC 9814 §4, freely mixed with the classical and ML-DSA signers. The parameters-MUST-be-absent rule is enforced across every format's AlgorithmIdentifier. **Certificate signing is shipped** — `pki.x509.sign` issues an SLH-DSA certificate in any of the twelve parameter sets, verified by OpenSSL. RFC 9909 (X.509), RFC 9814 (CMS).
- **ML-KEM in CMS** — *Shipped.* `pki.cms.encrypt` / `pki.cms.decrypt` produce and consume ML-KEM-512/768/1024 KEMRecipientInfo recipients end to end (RFC 9629 / RFC 9936), riding the FIPS 203 KEM encapsulation/decapsulation now in the crypto engine, with ciphertext-size and parameters-absent enforcement. A post-quantum recipient is an OID-keyed registry row exactly like an RSA one. RFC 9629 / RFC 9936.
- **ML-DSA in CMS** — *Shipped.* `pki.cms.sign` and `pki.cms.verify` sign and verify a CMS SignedData with the post-quantum ML-DSA-44/65/87 — pure mode, empty context, `parameters` absent, freely mixed with classical signers in one SignedData. The message-digest algorithm is held to each parameter set's security strength (SHA-512 by default, SHAKE256 optional; a below-strength digest is refused on both sign and verify). RFC 9882.
- **Composite (hybrid) signatures** — *Certification-path + CMS verification shipped; CMS signing shipped.* Certification-path validation (`pki.path.validate`) verifies composite ML-DSA signatures — the transition-period construction pairing a post-quantum ML-DSA with a traditional RSA / ECDSA / EdDSA so a certificate stays trustworthy if either primitive is later broken. All 18 draft-19 algorithm identifiers are registered under the `id-alg` arc and their AlgorithmIdentifier parameters-absent rule is enforced across every format; verification reconstructs the domain-separated message representative, checks the ML-DSA component (with the composite Label as its context) and the traditional component independently, and accepts only when **both** pass — an OR would be the downgrade this construction exists to prevent. The same combinator backs a composite-signed CRL / OCSP response. Fifteen algorithm arms verify today; the two brainpool-curve arms and the one SHAKE256-pre-hash arm are registered and parameters-guarded but fail closed to `path/unsupported-algorithm` at verify until their curve / digest is reachable. Composite ML-DSA CMS `SignerInfo` **signing and verification are shipped** — `pki.cms.sign` / `pki.cms.verify` sign and verify a SignedData whose signer uses a composite arm, the arm's pre-hash fixing the SignerInfo digestAlgorithm (draft-ietf-lamps-cms-composite-sigs) and both components signed and verified over the domain-separated preimage. X.509 composite *certificate* issuance is shipped — `pki.x509.sign` issues a certificate under a composite arm, signing both the ML-DSA and the traditional component over the certificate's `tbsCertificate`. draft-ietf-lamps-pq-composite-sigs. Composite KEMs follow — draft-ietf-lamps-pq-composite-kem.
- **Stateful hash-based signatures (verify-only)** — *Shipped (HSS/LMS); XMSS Planned.* `pki.shbs.verify` / `verifyLms` verify HSS/LMS signatures (RFC 8554, carried by RFC 9802 X.509 and RFC 9708 CMS), proven against the RFC 8554 Appendix F and full NIST ACVP known-answer suites across every SP 800-208 parameter set. Pure public-input SHA-256 / SHAKE256 hashing, a data-driven typecode registry, fail-closed. **XMSS / XMSS^MT verification** remains *Planned* (RFC 8391 ships no known-answer vector and NIST ACVP does not yet cover XMSS, so it awaits an authoritative interoperability vector), as does **automatic HSS/LMS verification inside `pki.path.validate`** (awaiting a real HSS-signed certificate to prove the path wiring end to end; operators verify today by handing the raw blobs to `pki.shbs.verify`). Signing is deliberately out of scope — stateful-key reuse is catastrophic and belongs in an HSM. RFC 8554 / RFC 8391 / NIST SP 800-208.
- **HPKE** — *Shipped (classical DHKEM); PQ KEMs Planned.* `pki.hpke` implements the RFC 9180 base construction — `setupS`/`setupR` + `seal`/`open`/`export` over the DHKEM suites P-256, P-521, X25519, and X448, HKDF-SHA256 / HKDF-SHA512, all three AEADs plus export-only, and all four modes — the composition ECH / MLS / OHTTP build on, each proven against the RFC 9180 Appendix A vectors. Pure composition over the native crypto engine, no ASN.1. **DHKEM(P-384) and HKDF-SHA384** are RFC-registered but Appendix A ships no known-answer vector for them, so they are not offered — a request fails closed — until an authoritative KAT exists; both admit as a one-row registry addition then. **ML-KEM and the X-Wing PQ-hybrid as HPKE KEMs** likewise remain *Planned*: their HPKE bindings are specified only by moving drafts with no stable interoperability vector, so the KEM registry is shaped to admit them as a data-row addition once those specs stabilize. RFC 9180.
- **FN-DSA** — *Under design.* Registry rows and the parameters-absent treatment reserved so the FIPS 206 lattice signature becomes a data-row addition once the standard and the runtime land. FIPS 206 (draft).
- **HQC** — *Under design.* Registry rows reserved for the code-based KEM NIST selected in March 2025 as a second-track standard alongside ML-KEM — a hedge on different mathematical foundations (syndrome decoding rather than structured lattices) against a lattice break. Encapsulation/decapsulation and the X.509 / CMS KEMRecipientInfo bindings become a data-row addition once NIST's draft standard and a runtime implementation land. NIST PQC (draft standard forthcoming).
- **EdDSA** — *Shipped (verify).* Ed25519/Ed448 signature verification is wired into certification-path validation with the issuer-key/signature-algorithm consistency check; X25519/X448 key agreement and EdDSA signing follow. RFC 8410 / RFC 8032.
- **KEM usage discipline** — *Planned.* NIST's normative rules for using a KEM — mandatory re-encapsulation with implicit rejection on decapsulation, the shared secret consumed only through a KDF, and approved combiners for hybrids — govern the ML-KEM decapsulation path (the RFC 9629 KEMRecipientInfo decrypt already targeted) and the composite / X-Wing hybrid combiners, so KEM key establishment follows the standardized construction rather than an ad-hoc one. NIST SP 800-227.
- **Additional PQC signatures (the NIST on-ramp)** — *Forward-watch.* Beyond the main-track winners, NIST is standardizing additional, mostly non-lattice signatures as a diversity hedge (SQIsign, MAYO, UOV, HAWK, FAEST, CROSS and others, in multi-round evaluation). None has an assigned OID or a drafted standard yet; the OID-keyed registry is shaped to admit the eventual winner as a data-row addition, not a rewrite. NIST PQC Additional Digital Signatures project.
- **Post-quantum transition timeline** — *Forward-watch.* NIST's migration guidance deprecates quantum-vulnerable RSA/ECC after 2030 and disallows it after 2035, endorsing hybrid / dual-algorithm operation through the transition — the deprecation clock the toolkit's PQC-first defaults and its composite/hybrid surfaces are built against, tracked so algorithm-policy gates can enforce it as the dates land. NIST IR 8547 (draft) / SP 800-131A Rev. 3 (draft).
- **Standards conformance (the shipped primitives).** The classical and hash primitives the engine already runs each map to their governing NIST standard, resolved through the same OID registry the parsers read so the compliance mapping stays data-driven: ECDSA / EdDSA / RSA verification to **FIPS 186-5** — including the §6.4.2 reject of ECDSA r/s outside [1, n-1] that defeats the zero-signature forgery class (CVE-2022-21449) — over the **SP 800-186** named curves; SHA-2 / SHA-3 / SHAKE to **FIPS 180-4** / **FIPS 202**; AES-GCM to **FIPS 197** + **SP 800-38D**; HMAC to **FIPS 198-1**; the HPKE key-schedule / key-establishment KDF to **SP 800-56C**; and the HSS/LMS stateful hash-based verification already shipped to **SP 800-208**.

## Revocation and transparency

- **OCSP** — *Shipped.* `pki.ocsp` is the full RFC 6960 request/response surface: `buildRequest` (one or many `{ cert, issuer }` queries, CertID under SHA-1 or SHA-2, optional RFC 9654 nonce, optional requestor signature, the RFC 5019 lightweight profile), `sign` (a signed BasicOCSPResponse over the exact ResponseData DER, the issuing CA or a delegated responder, good / revoked / unknown status, any `pki.cms.sign` algorithm including the post-quantum ML-DSA and SLH-DSA sets), `buildErrorResponse` (the unsigned §2.3 error), and `verify` (fail-closed relying-party verification: CertID binding under the CertID's own hash, responder authorization, signature over `tbsResponseDataBytes`, currency, and RFC 9654 nonce echo). Verification composes the same hardened responder-authorization, signature, and currency core as `pki.path.ocspChecker` — there is no weaker second verify path — also exposed lower-level as `pki.path.verifyOcspResponse`; both directions are cross-checked against `openssl ocsp`. RFC 6960 / RFC 9654 / RFC 5019.
- **Certificate Transparency SCTs** — *Parse, verify, encode, sign, and log-list trust shipped.* `pki.ct.parseSctList` decodes the SCT-list extension to per-SCT log id, exact timestamp, named algorithm, and raw signature (unknown versions preserved opaque); `pki.ct.reconstructSignedData` rebuilds the exact digitally-signed preimage; `pki.ct.verifySct(entry, sct, logKey)` verifies an SCT's signature against a log's public key per RFC 6962 §3.2 — reconstructing the signed data, routing an ECDSA signature through the shared strict DER-conformance gate before conversion to the raw form the crypto engine expects, and verifying through `pki.webcrypto`. The producing side: `pki.ct.encodeSctList(scts)` builds the SCT-list extension value byte-for-byte (the exact inverse of `parseSctList`), and `pki.ct.signSct(entry, logKey)` performs a log's signing step — rebuilding the same signed-data preimage the verifier hashes and signing it with the log's ECDSA-P-256 / RSA key to yield a v1 SCT `verifySct` accepts. The trust surface: `pki.ct.parseLogList(json)` ingests the CT log-list JSON (the `log_list.json` browsers consume) into constraint-carrying trusted logs — recomputing each log's id as SHA-256 of its key and refusing a stated id that disagrees (a swapped key or flipped id, §3.2), and decoding the log state + temporal interval — and `pki.ct.verifySctWithLogList(entry, sct, logList)` resolves the log key from the SCT's log id, enforces the state (usable/qualified/readonly trusted; retired only before retirement; pending/rejected refused) and the temporal-interval window (the covered cert's notAfter), then delegates the signature check to `verifySct`. `pki.ct.verifyLogListSignature(json, signature, publicKey)` verifies the detached `log_list.sig` over the raw log-list bytes against a caller-pinned signer key (RSASSA-PKCS1-v1.5/SHA-256, an EC P-256 arm for future-proofing; forgeable-key defenses fail closed) — cross-checked against `openssl dgst -verify`, completing the offline log-list trust chain (verify the signature, then `parseLogList`). *Planned next:* a live-fetch client for the log list. RFC 6962.
- **Merkle transparency proofs** — *Proof core shipped; envelope under design.* The RFC 6962 / RFC 9162 inclusion and consistency proof-verification core shipped as `pki.merkle` — `leafHash` / `nodeHash` / `emptyRootHash` plus `verifyInclusion` / `verifyConsistency`, fail-closed with a constant-time root compare over sync SHA-256, transport-free. The C2SP checkpoint / signed-note / tile envelope layer (the static-CT-API tiled-log parsing that feeds proofs into this core) remains under design. One core serves a static-CT log client, Merkle Tree Certificate verification, transparency-log inclusion proofs, and the shipped SCT surface. RFC 6962 / RFC 9162; C2SP static-ct-api / tlog-tiles / tlog-checkpoint.
- **Merkle Tree Certificates** — *Under design.* Verification of the post-quantum-oriented certificate format (compact, transparency-log-anchored) that major CAs and browsers are piloting, composing the Merkle-proof core. draft-ietf-plants-merkle-tree-certs.
- **Partitioned / sharded CRLs** — *Shipped.* The `cRLDistributionPoints` / `freshestCRL` extension decoders plus the RFC 5280 §6.3.3 Distribution Point ↔ Issuing Distribution Point correspondence in `pki.path.crlChecker`: a partitioned CRL whose critical IDP shares an identically-encoded name with one of the certificate's distribution points, carries no reason restriction, is current, and verifies, establishes non-revocation for its shard. Non-corresponding, reason-restricted, delta, or unverifiable shards fail closed to undetermined; a listed serial reports revoked regardless. Reason-shard mask accumulation and delta merge remain *Planned* (fail-closed today). RFC 5280 §5 / §6.3.
- **Trust-store ingestion (`pki.trust`)** — *Shipped.* `pki.trust.parseCertdata` / `parseCcadbCsv` ingest the Mozilla/NSS `certdata.txt` object stream and the CCADB certificate-records CSV into constraint-carrying trust anchors — per-purpose trust bits (only a trusted delegator grants a purpose) and distrust-after dates the bare root list omits — and `pki.path.validate` enforces them via `opts.checkPurpose` (`path/distrusted-after`, `path/purpose-not-trusted`). The CCADB per-root name-constraint overlay and live fetching remain *Planned*.

## Timestamping — RFC 3161

- **Time-Stamp Protocol** — *Shipped.* `pki.tsp.sign(messageImprint, tsa, opts)` creates an RFC 3161 TimeStampToken — a CMS SignedData (over `pki.cms.sign`) whose content is a `TSTInfo` carrying the message imprint, TSA policy, serial number, and `genTime` (with optional accuracy / nonce / ordering), plus the RFC 3161 §2.4.2 / RFC 5816 signing-certificate (ESSCertIDv2) attribute binding the token to the TSA certificate; `pki.schema.tsp.parseToken` decodes one. `pki.tsp.request` / `pki.tsp.parseRequest` build and parse the `TimeStampReq` a client sends (message imprint, requested policy, nonce, certReq), and `pki.tsp.response` / `pki.tsp.parseResponse` build and parse the `TimeStampResp` a TSA returns — a granted status wrapping a token, or a rejection carrying a PKIStatus and failure info — with the §2.4.2 status↔token coupling enforced in both directions. `pki.tsp.verify(token, data, opts)` verifies a token: the CMS signature over the exact signed bytes, the message imprint recomputed from the original data, the id-ct-TSTInfo content type, the ESSCertID(V2) binding of the token to the TSA certificate, the §2.3 requirement that the TSA certificate carry a critical extendedKeyUsage of exactly id-kp-timeStamping, the request nonce when one was used, and — when a trust anchor is supplied — full certification-path validation of the TSA certificate at the token's `genTime`; a well-formed token failing any check is a fail-closed `{ valid: false }` verdict with a typed reason code. A message layer, not an HTTP client — the builders emit the byte payloads the RFC 3161 §3 transports carry. RFC 3161 / RFC 5816.

## Attribute certificates — RFC 5755

- **Attribute certificates** — *Shipped (v2 parse + value decoding).* `pki.schema.parse` detects and decodes an RFC 5755 v2 attribute certificate — the `AttributeCertificateInfo` (holder, issuer, validity, attributes, extensions), the signature algorithm, and the signature — surfacing the raw `acinfo` bytes a signature verifier hashes. The RFC 5755 §4.4 attribute values (role, clearance, service-authentication/access identity, group, charging identity) and §4.3 extensions (audit identity, target/proxy information, no-revocation-available, AA controls) are decoded to structured values additively (a `decoded` field alongside the raw bytes), with every embedded GeneralName validated, an unrecognized type preserved opaque, and a malformed recognized value failing closed. The obsolete v1 profile is recognized and declined with a typed error. Attribute-certificate *building* remains planned. RFC 5755.

## Enrollment and lifecycle protocols

Certificate lifecycle management is absent from the toolkit this library replaces; it is a first-class goal here.

- **ACME** — *Shipped (message layer).* `pki.acme` is the RFC 8555 message layer (updated by RFC 8737 tls-alpn-01, RFC 8738 IP identifiers, and RFC 9773 ARI) over the new `pki.jose` JWS surface: resource-object validators (closed status enums, conditional-required fields, unknown fields ignored), the three §7.1.6 state machines, the request builders (newAccount + External Account Binding, newOrder + `replaces`, finalize with CSR identifier-set match and account-key-reuse rejection per §11.1, challenge responses, deactivation, revokeCert in both key modes, the keyChange nested JWS, POST-as-GET), the http-01 / dns-01 / tls-alpn-01 challenge computations, the dns/ip identifier validators, and the ARI certID (serial sign-padding preserved). It is a message layer, not an HTTP client — the transport is injectable and every codec is fail-closed. The thin `node:https` client that drives the nonce/POST-as-GET/retry flow over a live directory is *Planned*. RFC 8555 / RFC 8737 / RFC 8738 / RFC 9773.
- **JOSE** — *Shipped.* `pki.jose` is the RFC 7515 Flattened JWS and RFC 7638 JWK-thumbprint layer ACME composes: a strict base64url codec, a bounded duplicate-member-rejecting JSON reader, profiled `sign`/`verify`, an algorithm registry spanning ES/RS/PS/EdDSA/ML-DSA that binds each `alg` to its key type, and canonical thumbprints (RFC 8037 OKP, RFC 9964 AKP). RFC 7515 / RFC 7518 / RFC 7638 / RFC 8037 / RFC 9964.
- **EST** — *Shipped (client codecs).* `pki.est` is the transport-agnostic Enrollment over Secure Transport client surface (RFC 7030, updated by RFC 8951 and RFC 9908): the base64 transfer codec (blind to Content-Transfer-Encoding), the `multipart/mixed` splitter, the certs-only and serverkeygen response validators layered over the CMS parser, the enroll-attribute builders (channel-binding challengePassword, out-of-band key identifiers, SMIMECapabilities, the RFC 9908 template-priority plan), and the HTTP response classifier (202 Retry-After surfaced, 204/404-on-csrattrs a "none available" verdict). Its companion `pki.schema.csrattrs` parses the CSR Attributes wire format (RFC 8951 §3.5 / RFC 9908). The thin `node:https` client with the TLS-floor / auth-ordering / redirect defaults is *Planned*. RFC 7030 / RFC 8951 / RFC 9908.
- **Certificate request messages (CRMF)** — *Parse and build shipped.* `pki.schema.crmf.parse` decodes an RFC 4211 CertReqMessages — the request body CMP and EST enrollment carry — into the requested-certificate template, proof-of-possession, and registration controls, surfacing the raw CertRequest bytes a proof-of-possession verifier hashes. `pki.crmf.build` assembles one — a CertTemplate of the requested certificate fields plus a POPOSigningKey proof of possession signed with the private half of the requested key (or a raVerified proof), across every signature algorithm the registry resolves — carrying requested extensions, registration controls, and regInfo, in a single message or a batch. `pki.cmp.build` wraps them in the CMP `ir` / `cr` / `kur` request body (see below).
- **CMP** — *Parse and build shipped.* `pki.schema.cmp.parse` decodes an RFC 9810 PKIMessage — the protected transport envelope CMP enrollment, revocation, confirmation, and support exchanges ride — into its header, its 27-arm body (certificate requests via the CRMF parser, encrypted certificates via CMS, response / revocation / confirmation / error / support / polling arms structural, the rest raw), protection inputs, and extra certificates, surfacing the exact byte slices an external verifier reconstructs the protected part from. `pki.cmp.build` assembles a protected PKIMessage — a header (sender / recipient, including the anonymous NULL-DN, plus transaction metadata) and a single-arm body across both directions: request-side `ir` / `cr` / `kur` (a CertReqMessages via `pki.crmf.build`), `p10cr` (a PKCS#10 CertificationRequest), `certConf` / `pollReq` / `genm` / `rr`; and CA/responder-side `ip` / `cp` / `kup` / `ccp` (a CertRepMessage of CertResponse entries, each a PKIStatusInfo and, under a granting status, a certifiedKeyPair), `rp` (revocation response), `genp`, `error`, `pollRep`, `krp` (key-recovery response), `pkiconf`. Protection is a sender-key signature (any registry algorithm, resolved from the signer certificate) or a PBMAC1 shared-secret HMAC (RFC 9481 / 9579), computed over the exact ProtectedPart DER; the protectionAlg is derived rather than caller-set, and the protection self-verifies before the message is returned, so every message round-trips through the parser. HTTP transfer (RFC 9811), the private-key-transport / KEM protection flavors, the lightweight-profile (RFC 9483) validation rules, and incoming MAC / signature verification are *Planned*. RFC 9810 / RFC 9483 / RFC 9481.
- **SCEP** — *Under design.* Simple Certificate Enrollment Protocol for MDM and network-device enrollment. RFC 8894.
- **CMC** — *Under design.* Certificate Management over CMS, for federal/PIV ecosystems. RFC 5272 / 5273 / 5274.

## High-level builder API

- **Safe-by-default builders** — *Targeted.* Ergonomic certificate, CSR, and CMS builders that accept strings and plain objects, with first-class PEM input and output and sensible extension helpers. The natural, easy path is the safe and correct one. Low-level ASN.1 structures remain available but are opt-in, and every public API member is public — no legitimate use is blocked by visibility.

## Messaging

- **S/MIME** — *Signing, verification, encryption, and compression shipped* (`pki.smime`), over the shipped ESS attribute decoders (`pki.schema.smime`). `pki.smime.sign(content, signers, opts)` assembles a signed S/MIME message in both RFC 8551 forms — `multipart/signed` (clear-signed, a detached CMS signature alongside the readable content) and `application/pkcs7-mime; smime-type=signed-data` (opaque) — and `pki.smime.verify` unwraps and verifies both, recomputing over the same RFC 8551 §3.1.1 canonicalizer the signer used (a shared `lib/mime.js`), so a transport that re-wraps line endings still verifies and a tampered part fails. `pki.smime.encrypt(content, recipients, opts)` envelopes a MIME entity as an opaque `application/pkcs7-mime` message and `pki.smime.decrypt` opens one — `smime-type=authEnveloped-data` (AES-GCM, confidentiality and integrity, the default) or `smime-type=enveloped-data` (AES-CBC, confidentiality only, so `decrypt` reports `authenticated: false`, the §3.3 no-integrity caveat); the `smime-type` is derived from the CMS body, not the header. The crypto is entirely `pki.cms.sign` / `verify` / `encrypt` / `decrypt`, so any RSA / RSASSA-PSS / ECDSA / EdDSA / ML-DSA / SLH-DSA signer and any RSA-OAEP / ECDH / X25519 / X448 / AES-KW / PBKDF2 / ML-KEM recipient carries through; decryption is fail-closed and oracle-free; bidirectionally interoperable with `openssl smime` / `openssl cms`. `pki.smime.compress` / `pki.smime.decompress` add the opaque `application/pkcs7-mime; smime-type=compressed-data; name=smime.p7z` frame (RFC 8551 §3.6) over the CMS CompressedData layer — a size transform with no integrity/confidentiality (§2.4.5), decompress-bomb-bounded. The RFC 5035 SigningCertificate(V2) and SMIMECapabilities signed-attribute decoders continue to surface a SignerInfo's signing-certificate binding + capability list. *Planned next:* header protection (RFC 8551 §3.3 `message/rfc822` wrapping). RFC 8551, RFC 5083, RFC 5035, RFC 3274.

## Attestation and supply chain

Path validation's fastest-growing consumers are attestation chains, and the Node ecosystem has no native, zero-dependency layer for any of them.

- **WebAuthn / passkey attestation** — *Shipped (attestation-statement verification).* `pki.webauthn.parseAttestationObject` decodes the attestation object + authenticatorData + COSE credential key over the strict `pki.cbor` codec, and `pki.webauthn.verify` cryptographically verifies each defined attestation-statement format — packed, tpm, android-key, apple, fido-u2f, none — checking the signature and each format's structural bindings (the x5c leaf key, the apple nonce, the tpm `certInfo` Name/`extraData` over the `pubArea`, the android `KeyDescription`, the fido-u2f `verificationData`), binding every attestation certificate key to the credential key, and enforcing each leaf's certificate requirements. The credential-key evaluation covers the full WebAuthn COSE algorithm set including the RFC 9864 fully-specified identifiers (ESP256/384/512, Ed25519) and Ed448, validates the public-key point on-curve, rejects the compressed EC point form, and requires a minimally-encoded DER ECDSA signature. Verified against the official W3C WebAuthn Level 3 test-vector suite and captured attestations across every format. Chaining the returned trust path to a pinned root — and resolving a root from an authenticator's aaguid via the FIDO Metadata Service — remains *Planned*, re-openable behind a caller-supplied root set or an `opts.mdsBlob`; the verifier surfaces the x5c chain for `pki.path.validate`. W3C WebAuthn Level 3, RFC 9864.
- **Sigstore / provenance verification** — *Shipped (offline bundle verification).* `pki.sigstore.verifyBundle` verifies a Sigstore bundle (the `npm publish --provenance` artifact) offline against caller-pinned trust: the DSSE signature over its PAE under the Fulcio leaf key, the ephemeral Fulcio chain validated as of the Rekor log time through the toolkit's own path validator, the RFC 9162 inclusion proof folded to a Rekor-signed root through the Merkle-proof core, the log entry bound to this exact signature, and the in-toto SLSA subject digest — with no dependency tree of its own. The sigstore TUF trust-root client, Rekor v2 tiled logs, the `message_signature` arm, and embedded-SCT verification remain *Planned* (each re-openable behind a caller opt-in).

## Alternative encodings

- **C509 CBOR certificates** — *Parsing and encoding shipped (both certificate types, type-3 DER round-trip both directions).* `pki.schema.c509.parse` decodes a C509 certificate (draft-ietf-cose-cbor-encoded-cert) in both forms — natively-signed and the CBOR re-encoding of a DER X.509 v3 certificate — over the strict deterministic-CBOR `pki.cbor` codec (RFC 8949 §4.2). A type-3 certificate is inverted back to the original DER byte-for-byte (de-compressing the EC point, re-emitting each field as canonical DER, re-wrapping the ECDSA signature), so the original signature still verifies and the result round-trips through `pki.schema.x509.parse`; a field outside the reconstruction's covered set fails closed. `pki.schema.c509.encode` is the producing inverse: a DER X.509 v3 certificate forward-transforms to a compact type-3 C509 — compressing the EC point to its sign-of-y marker, a MAC-address commonName to a tag-48 EUI, a lone keyUsage to its integer shortcut, and re-expressing the ECDSA signature as a fixed-width r‖s — and self-verifies the transform reproduces the source DER byte for byte; a `parse` result re-emits its native array. It decodes and encodes CBOR, not DER, so it is an explicit-call surface, not auto-routed by `pki.schema.parse`. Underpinning it, `pki.cbor.build` is a deterministic-CBOR encoder — the byte-exact inverse of the strict decoder. The deferred per-extension value inversions remain *Planned*.

## Assurance and interoperability

- **Certificate linting** — *Shipped (RFC 5280 profile + a representative CABF TLS BR subset), with the `pki inspect / lint / convert / verify` CLI; the full CABF corpus, S/MIME BR, and CRL/OCSP profiles are Targeted.* `pki.lint.certificate(pem | der | parsed, opts)` is a profile-conformance lint engine over the strict codec and schema: a declarative rule set that emits graded advisory findings — each with a stable id, a severity (fatal > error > warn > notice), a source, a spec-clause citation, and a message. Unlike the rest of the toolkit its advisory result surface never throws on the data path (hostile bytes become a `fatal` `lint/unparseable` finding carrying the strict parser's code); only config-time misuse raises a typed `LintError`. `rules(profile)` and `profiles()` enumerate the registry. v1 covers the RFC 5280 certificate profile (serial sign/length, validity ordering, keyCertSign coherence, extension criticality — basicConstraints/nameConstraints/policyConstraints/inhibitAnyPolicy must be critical and keyUsage should be, nameConstraints CA-scope, unknown critical extensions, empty-subject SAN, SKI/AKI advisories including the end-entity subjectKeyIdentifier) and a representative CABF TLS BR subset (SAN required, CN-in-SAN, dNSName syntax, serverAuth EKU, weak keys, the SC081v3 reducing validity ceiling (398 → 200 → 100 → 47 days, gated to issuance date)). Because this toolkit parses strict-DER, several profile violations a lenient linter reports specifically are rejected at parse and surface as one `lint/unparseable`. Pre-issuance linting is now expected CA practice, and no JavaScript linter existed.
- **Interoperability gates** — *Shipped (OpenSSL + NSS); broader PKCS#12 matrix Targeted.* Cross-tool round-trip and parse checks against OpenSSL and NSS run as containerized release-acceptance gates; the Windows CAPI and macOS Keychain PKCS#12 cross-checks land with PKCS#12 building.
- **Fuzzing** — *Shipped.* A coverage-guided fuzz corpus against the ASN.1 codec and every parse path runs continuously — per-PR and nightly across a per-target matrix, each harness holding the contract that hostile bytes may only parse or throw a typed `PkiError`.
- **Independent security review** — *Planned.* A published security assessment of the parser and every crypto path as a release deliverable.

## Documentation

- **Source-generated reference and runnable examples** — *Shipped; example coverage expanding.* API documentation is generated from per-namespace `@module` and per-primitive `@primitive` source comment blocks and validated in continuous integration on the shipped runtime, so it cannot drift from the code. Example coverage keeps growing toward the common tasks — signing with your own CA, parsing a certificate, building a CSR with a SAN, PEM conversion — that otherwise generate repeated questions.

**7. Post-quantum-specific pitfalls** (design-class; implementation hazards, not all with a single clean CVE)

| Hazard | Representative real-world finding | Design mitigation |
|---|---|---|
| KEM decapsulation failure handling | Fujisaki–Okamoto implicit-rejection requirement (FIPS 203) | ML-KEM decapsulation always runs the re-encryption check and returns the implicit-rejection secret on mismatch — never branches on decapsulation success/failure |
| Secret-dependent timing | **KyberSlash 1/2** (secret-dependent division timing in Kyber/ML-KEM decaps, 2024); **Clangover** (CVE-2024-37880 — compiler turned constant-time source into variable-time binary; CVSS unverified) | No secret-dependent division/branch/memory access; assert constant-time survives the JIT/optimizer; avoid table lookups on secret indices |
| ML-DSA context-string handling | Domain-separation context omission / pure-vs-pre-hash confusion | Always bind the context string; reject context > 255 bytes; keep pure and pre-hash modes distinct and explicit |
| Hybrid / composite downgrade | Accepting a composite signature/KEM when only the classical (or only the PQC) component verifies | Composite verification requires **all** component signatures/KEMs to pass; no negotiation down to a single component |
